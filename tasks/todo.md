## Tactical decisions (please review before approval)

### D1. Doc-ID strategy in migration
**Decision:** preserve legacy doc ID as new doc ID (`announcements/<sameId>`).
- ✅ Idempotent re-runs: re-running the script over already-migrated docs is a no-op
- ✅ Easy audit: legacy ID `xyz` → new doc `xyz` with `migratedAt` timestamp
- ⚠️ Cosmetic: new doc IDs were auto-generated by CF; migrated ones inherit legacy format (mixed in production)
- ❌ Alternative (new auto-id + `legacyId` field): forces a second mapping table for verification, more work, no clear win

### D2. Event sender for migrated docs
**Decision:** synthesize `sender = { uid: 'migration', email: 'migration@thegreenhaven' }`.
- Legacy events have `createdBy` (admin email) or nothing — not always present
- Migrated marker is more useful than fake admin attribution
- Production-shape docs keep their real sender (already correct)

### D3. Audience normalization for legacy events
Legacy `communityEvents.{building}` field values seen in code/data: `'all'`, `'rooms'`, `'nest'`, possibly aliases `'old'`/`'new'`.

**Decision:**
```
'all'     → 'all'
'rooms'   → 'rooms'
'nest'    → 'nest'
'old'     → 'rooms'   (via BuildingConfig.ALIASES)
'new'     → 'nest'
<other>   → 'all'     (default — log warning, no migration aborted)
```
Rationale: keeps the audience valid set per the C4 rule. Falling back to `'all'` is safe (matches legacy behavior — pre-C4 events were public).

### D4. Event edit affordance during S2
After migration, legacy `communityEvents/{id}` docs become "dead duplicates" — admin shouldn't edit them via the legacy path anymore (those edits won't reach the new collection and tenants won't see them).

**Three options — pick one:**

| Option | What it means | Lines changed | Tradeoff |
|--------|---------------|---------------|----------|
| **A (recommend)** | Disable admin event edit in S2; show toast "การแก้ไขกิจกรรมจะรองรับใน S3" + hide ✏️ button | ~10 lines + 1 button hide | Edit UX gone for 1 session; minimal scope |
| B | Build new `updateAnnouncement` CF + ~10 unit tests + wire admin edit through it | ~150 lines + CF deploy | Full edit works; adds CF surface S3 would inherit |
| C | Leave legacy edit path alive; admin warned via toast "นี่คือ legacy edit — tenant อาจไม่เห็น" | ~5 lines | Confusing UX; defeats migration purpose |

I recommend **A** for S2 scope discipline. B is a clean S3 addition (or S2 bonus if you want).

### D5. Migration script run order vs deploy order
Migration script writes to `announcements/{id}` via Admin SDK (bypasses rules). Tenant code flip must happen AFTER migration completes — otherwise tenants briefly see empty bell/feed (legacy reader dropped, new collection still partial).

**Decision:** strict serial deploy:
```
1. Push migration script (no behavior change — script is opt-in)
2. User runs `tools/migrate-to-announcements.js` (dry-run → review → --apply)
3. Verify in Firestore Console: announcements has N legacy notices + M legacy events with migratedAt
4. Then push tenant + admin code changes (drops legacy reads)
5. Vercel auto-deploys; live verify via Chrome MCP
```

## Migration script architecture (`tools/migrate-to-announcements.js`)

Single script, 2 phases, idempotent. Modeled on [tools/c4-backfill-legacy-announcements.js](tools/c4-backfill-legacy-announcements.js).

```text
PHASE 1 — broadcastMessages → announcements (type='notice')
  Read every doc in broadcastMessages/
  For each:
    if announcements/<sameId>.exists && has migratedAt → skip (idempotent)
    else build payload:
      type: 'notice'
      title, body, audience: copy from legacy (already canonical-shape — they're CF-written since Tier 1A)
      sender: copy from legacy
      sentAt: copy from legacy
      status: legacy.status || 'published'
      migratedAt: Timestamp.now()
    write announcements/<sameId> via .set(payload) (NOT merge — full doc since we own all fields)

PHASE 2 — communityEvents → announcements (type='event')
  Read every doc in communityEvents/
  For each:
    if announcements/<sameId>.exists && has migratedAt → skip
    else build payload:
      type: 'event'
      title: legacy.title
      body: legacy.description || legacy.title
      audience: normalize(legacy.building) per D3
      sender: { uid: 'migration', email: 'migration@thegreenhaven' }
      sentAt: Timestamp.fromDate(parse(legacy.createdDate || legacy.createdAt || legacy.date))
      status: 'published'
      eventDate: Timestamp.fromDate(parse(`${legacy.date}T${legacy.time || '00:00'}`))
      location: legacy.location || ''
      migratedAt: Timestamp.now()
    write announcements/<sameId> via .set(payload)

REPORT
  Print summary: N notices migrated, M events migrated, K skipped (already migrated), W warnings (audience fallback)
```

**Edge cases handled:**
- Legacy doc with corrupt `date` / `time` → eventDate falls back to `sentAt` (won't break query)
- Legacy doc lacking `building` → audience='all' fallback
- Concurrent admin write to legacy collection during migration → next re-run picks it up (idempotent)
- `--apply` partial failure (network) → next run resumes from where it stopped

## Tactical decisions to confirm BEFORE coding

### W1. Speed test implementation
Three viable options, increasing in accuracy + effort:

| Option | What | Accuracy | Effort | Risk |
|--------|------|----------|--------|------|
| **W1-A** `navigator.connection.downlink` | Browser-reported estimate (Network Information API). No fetch. | Low (browser-throttled estimate, often ±50%) | XS (1 line) | None — pure read |
| **W1-B** Fetch small payload + time | Download ~2MB asset from same-origin CDN, compute Mbps. Repeat for upload via POST. | Medium (single-stream, not multi-flow like Ookla) | S (50-80 LOC) | Low — but stream may be CPU-cached; need cache-bust + random asset selection |
| **W1-C** Embed open-source HTML5 speed test | LibreSpeed (open source) or Cloudflare radar. iframe or imported script. | High (multi-stream, server-side timing) | M (200+ LOC + server endpoint or third-party dependency) | Medium — bundle bloat (Tailwind LIFF perf cost), third-party trust |

**Recommendation: W1-B** — accurate enough for "is it usable today?" diagnostic, no third-party dep, ~50 LOC. Add an HTML asset at `/speed-test-blob.bin` (2MB random data, generated at build time + cached forever).

### W2. What to do with existing `roomWifi/*` prod docs
Per current rule, only admin (or matching room tenant) can read. After we remove reader code, they sit orphaned.

| Option | Action | Pro | Con |
|--------|--------|-----|-----|
| **W2-A** Delete now via Admin SDK script | Run a `tools/cleanup-roomWifi.js --apply` | Clean slate, no orphans | Permanent — if rollback needed, data is gone (admin entered SSID/passwords aren't recoverable from anywhere else) |
| **W2-B** Leave orphaned + drop rule | Remove the `match /roomWifi` block from firestore.rules; docs sit unreadable | Reversible (data still in Firestore); rule deletion is the actual security gate | Orphaned docs sit forever; small Firestore storage cost; admins might wonder where data went |
| **W2-C** Mark deprecated + cleanup later | Update doc + flag for future migration | Safest now | Defers cleanup; relies on next session remembering |

**Recommendation: W2-B** — drop rule block (closes any access path) + add cleanup migration script to repo (`tools/cleanup-roomWifi.js`, dry-run by default) without running it yet. Future session can `--apply` after confirming nothing was missed.
