## ⚠️ 4. Decide CSP enforcement strategy (planning, no rush)

**Why:** `vercel.json` ships `Content-Security-Policy-Report-Only` with `'unsafe-inline'` for both `script-src` and `style-src`. With unsafe-inline, the header **provides zero security signal** — every inline handler and `<style>` block passes. The current CSP is theater.

**Options:**
- **(a) Delete the CSP header.** If you're never going to enforce it, the report-only entry is dead infrastructure. Shrinks `vercel.json` by ~30 lines.
- **(b) Plan a Q-next refactor.** Audit confirmed ~2,051 inline `style="..."` blocks and a smaller number of inline handlers. To enable strict CSP without `unsafe-inline`, all inline styles must move to classes (Tailwind already partly migrated) and all inline handlers must move to event delegation (already done in dashboard).

**Recommendation:** Pick (a) for now (kill the dead header), revisit (b) when there's a security review or a real vulnerability surfaces. Don't leave it report-only forever — it falsely implies you've thought about CSP.
