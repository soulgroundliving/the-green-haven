## 🚨 2. ⛔ DO NOT disable Anonymous auth (yet) — depends on a refactor

**Tested 2026-04-27, found broken:** disabling Anonymous breaks bills + every per-room RTDB read in tenant_app. **Re-enabled at Firebase Console.**

**Why:** the LIFF flow uses Anonymous UID as the seat that custom claims attach to:
1. `tenant_app.html:140` calls `signInAnonymously()` → gives the visitor an anon UID
2. LIFF init fires → captures the LINE userId
3. `linkAuthUid` CF receives `(anon-UID Bearer, lineUserId body)` → calls `setCustomUserClaims(anonUid, {room, building})`
4. Client `getIdToken(true)` picks up the new claims → RTDB rule `auth.token.room == $room` finally passes
5. Bills + maintenance + housekeeping queries succeed

If Anonymous is off, step 1 fails → there's no UID to claim → step 3 has nothing to update → bills don't load. New banner in tenant_app surfaces this (`auth/admin-restricted-operation`) so it's not a silent break, but the right fix is to remove the dependency.

**Refactor required before this can be disabled:**
- New CF `liffSignIn` that takes a LIFF idToken (verify via `https://api.line.me/oauth2/v2.1/verify`) → look up `liffUsers/{lineUserId}` (must be `approved`) → mint a custom token via `admin.auth().createCustomToken(uid, {room, building})` and return it
- `tenant_app.html` calls the CF instead of `signInAnonymously` + `linkAuthUid`, then `signInWithCustomToken(returnedToken)`
- Estimated effort: 1 session, including a careful LIFF-flow regression test on real LINE app

**Until then: keep Anonymous enabled.** The cleanup CF (item below) is still useful for housekeeping any one-off anon UIDs that piled up — it won't undo the mechanism.

## ⚠️ 4. Decide CSP enforcement strategy (planning, no rush)

**Why:** `vercel.json` ships `Content-Security-Policy-Report-Only` with `'unsafe-inline'` for both `script-src` and `style-src`. With unsafe-inline, the header **provides zero security signal** — every inline handler and `<style>` block passes. The current CSP is theater.

**Options:**
- **(a) Delete the CSP header.** If you're never going to enforce it, the report-only entry is dead infrastructure. Shrinks `vercel.json` by ~30 lines.
- **(b) Plan a Q-next refactor.** Audit confirmed ~2,051 inline `style="..."` blocks and a smaller number of inline handlers. To enable strict CSP without `unsafe-inline`, all inline styles must move to classes (Tailwind already partly migrated) and all inline handlers must move to event delegation (already done in dashboard).

**Recommendation:** Pick (a) for now (kill the dead header), revisit (b) when there's a security review or a real vulnerability surfaces. Don't leave it report-only forever — it falsely implies you've thought about CSP.

## 🚨 Incident log — 2026-04-27 evening

Two rule tightenings landed earlier in the session that broke tenant_app and had to be reverted:

1. **liffUsers read tightened to admin-only.** Symptom: tenants who cleared cache + re-entered via LINE LIFF lost their room link and saw the wrong room (or an empty UI). Root cause: `tenant_app.html:7626` reads `liffUsers/{lineUserId}` directly to discover their `status === 'approved'` and pull `room` + `building` — this read happens **before** `linkAuthUid` CF gets called (it's the lookup that triggers the CF call). With the rule denying tenant reads, the lookup returned permission_denied → `_callLinkAuthUid` was never invoked → no custom claims were set → every per-room RTDB read failed → bills disappeared. Reverted to `read: if isSignedIn()` and redeployed.

2. **complaints read scoped strictly to linkedAuthUid match.** Symptom: tenants saw zero of their old complaints (admin still saw them). Root cause: legacy complaint docs predate the `linkedAuthUid` field, so the equality check returned `null != uid` → false → denied. Relaxed the rule to also accept `resource.data.room == auth.token.room && resource.data.building == auth.token.building`, which makes legacy history visible to the tenant who actually owns the room (the room/building claims are still set by `linkAuthUid`). Create still requires `linkedAuthUid == auth.uid` so new docs are properly stamped.

The proper fix for both is the `liffSignIn` CF refactor mentioned in section 2 above — it would let us tighten `liffUsers` to admin-only safely (the CF would be the only Firestore reader). Do those together, not piecemeal.
